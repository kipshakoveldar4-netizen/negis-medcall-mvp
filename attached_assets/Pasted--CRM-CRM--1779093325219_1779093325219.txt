Нужно добавить отдельную CRM-воронку в раздел “Записи”.

Контекст:
Сейчас есть раздел CRM для отдела продаж.
Туда попадают лиды, с которыми работают агенты продаж.

Нужно:
В разделе “Записи” добавить отдельную CRM для клиентов, которых нужно записывать на услуги.

Важно:
Лиды из раздела “Записи” не должны пересекаться с лидами из раздела CRM отдела продаж.

Рекомендуемая архитектура:
Использовать одну таблицу leads, но добавить обязательное поле pipeline.

Поле:
pipeline text not null default 'sales'

Возможные значения:
- sales
- booking

Логика:
- текущий раздел CRM показывает только leads.pipeline = 'sales'
- раздел “Записи” показывает только leads.pipeline = 'booking'
- импорт лидов в разделе CRM создаёт leads с pipeline = 'sales'
- импорт лидов в разделе “Записи” создаёт leads с pipeline = 'booking'
- ручное создание лида в разделе CRM создаёт pipeline = 'sales'
- ручное создание лида в разделе “Записи” создаёт pipeline = 'booking'

Если нельзя менять существующую таблицу leads:
Создать отдельную таблицу booking_leads.
Но предпочтительно использовать поле pipeline, чтобы не дублировать логику.

1. Миграция Supabase

Добавить поле:

alter table leads
add column if not exists pipeline text not null default 'sales';

Добавить индекс:

create index if not exists leads_clinic_pipeline_created_idx
on leads (clinic_id, pipeline, created_at desc);

Опционально добавить check constraint:

alter table leads
add constraint leads_pipeline_check
check (pipeline in ('sales', 'booking'));

Если constraint уже есть или миграция падает, обработать аккуратно.

2. Раздел CRM отдела продаж

В текущем CRM:
Все запросы к leads должны фильтровать:

.eq('pipeline', 'sales')

Создание лида:
pipeline: 'sales'

Импорт лидов:
pipeline: 'sales'

Массовые действия:
дополнительно фильтровать:
.eq('clinic_id', clinicId)
.eq('pipeline', 'sales')
.in('id', selectedIds)

3. Раздел “Записи”

В разделе “Записи” добавить вкладку или панель:
- Календарь
- CRM записей

Можно сделать tabs:
- Календарь записей
- Лиды на запись

В “Лиды на запись” показать таблицу как в CRM:
- чекбокс
- имя
- телефон
- источник
- статус
- ответственный
- дата создания
- действия

Но данные брать только:

leads.pipeline = 'booking'

4. Создание лида в “Записи”

Добавить кнопку:
“Новый лид на запись”

Поля:
- Полное имя
- Телефон
- Email
- Компания/клиника/комментарий, если нужно
- Источник
- Статус
- Ответственный
- Комментарий

При insert:
{
  clinic_id,
  pipeline: 'booking',
  full_name,
  phone,
  email,
  source,
  status_id,
  assigned_to,
  comment
}

Важно:
- full_name обязателен;
- если full_name пустой, но есть phone: full_name = `Клиент ${phone}`;
- assigned_to должен быть правильным UUID согласно FK;
- не отправлять пустую строку в UUID поля, использовать null.

5. Импорт файла в “Записи”

В разделе “Записи” добавить импорт файла именно для booking pipeline.

Импорт должен:
- читать CSV/Excel;
- нормализовать колонки;
- создавать leads с pipeline = 'booking';
- не добавлять их в CRM продаж.

Поддержать колонки имени:
- full_name
- name
- имя
- фио
- клиент
- пациент

Поддержать колонки телефона:
- phone
- телефон
- номер
- whatsapp
- contact
- mobile

Если имени нет, но есть телефон:
full_name = `Клиент ${phone}`

Если нет имени и телефона:
строку пропустить и показать ошибку.

6. Создание записи из booking lead

При открытии лида в “Записи” должна быть кнопка:
“Создать запись”

Поля:
- услуга
- агент/специалист
- дата
- время
- комментарий

После нажатия:
- создать запись в bookings;
- связать booking.lead_id = lead.id;
- обновить статус лида на “Записан”, если такой статус есть;
- клиент должен появиться в ресепшне на выбранную дату.

7. Раздел “Ресепшн”

Ресепшн должен показывать bookings, а не leads напрямую.

То есть поток такой:
booking lead -> создать booking -> booking появляется в ресепшне

Лиды pipeline='booking' не должны сами по себе попадать в ресепшн, пока из них не создана запись.

8. Статусы

Нужно разделить статусы, чтобы не путать продажи и запись.

Вариант А:
В lead_statuses добавить pipeline:
- sales
- booking

Тогда:
- CRM продаж показывает статусы pipeline='sales'
- CRM записей показывает статусы pipeline='booking'

Миграция:
alter table lead_statuses
add column if not exists pipeline text not null default 'sales';

create index if not exists lead_statuses_clinic_pipeline_idx
on lead_statuses (clinic_id, pipeline, sort_order);

Вариант Б:
Оставить общие статусы, но это хуже.

Рекомендуемый вариант:
Добавить pipeline в lead_statuses.

Статусы для sales:
- Новый
- В работе
- Думает
- Отказ
- Записан

Статусы для booking:
- Новый
- Нужно записать
- Записан
- Недозвон
- Отмена

owner:
видит обе CRM

manager:

agent отдела продаж:

booking_agent:

receptionist:

Если сейчас ролей booking_agent нет, добавить роль:
booking_agent

11. Проверка

После реализации проверить:

Сценарий A:
- создать лид в CRM продаж;
- убедиться, что он не виден в Записи -> Лиды на запись.

Сценарий B:
- создать лид в Записи -> Лиды на запись;
- убедиться, что он не виден в CRM продаж.

Сценарий C:
- импортировать файл в CRM продаж;
- лиды получили pipeline='sales';
- они не видны в Записях.

Сценарий D:
- импортировать файл в Записи;
- лиды получили pipeline='booking';
- они не видны в CRM продаж.

Сценарий E:
- открыть booking lead;
- создать запись;
- перейти в Ресепшн;
- выбрать дату;
- клиент появился.

Сценарий F:
- ресепшн отметил “Пришёл”;
- отметка сохранилась после обновления.

Сценарий G:
- массовое редактирование в CRM продаж не влияет на booking leads;
- массовое редактирование в CRM записей не влияет на sales leads.

Ожидаемый результат:
В системе есть две независимые CRM-воронки:
1. CRM продаж
2. CRM записей

Они используют одну таблицу leads с разным pipeline или отдельные таблицы, если выбран такой вариант.
Лиды не пересекаются.
Из CRM записей можно создать запись, и она попадает в ресепшн.
