Нужно исправить ошибки Negis CRM после повторного мощного краш-теста.

Контекст:
Основной CRM сайт: https://www.negis.online
Админка Negis Control: https://admin.negis.online

Был создан отдельный тестовый набор:
- 10 ролей
- 25 агентов
- 40 услуг
- 10 статусов лидов
- 10 статусов записей
- 1000 лидов
- 1000 записей
- каждый агент получил ровно 40 записей
- все записи связаны с лидами через lead_id
- ресепшн обработал записи: 500 visited=true, 500 visited=false
- лиды после ресепшна не пропали

Нагрузка выдержана, но UI всё ещё работает неправильно.

Нужно исправить обязательно:

1. Исправить отображение имён лидов на /sales

Сейчас на /sales лиды отображаются с именем “—”.

Реальная таблица leads использует поле:

full_name

Нужно полностью убрать старую логику:

first_name
last_name

И везде использовать:

full_name

Проверить:
- список лидов;
- поиск по имени;
- открытие лида;
- редактирование лида;
- создание нового лида;
- импорт/экспорт лидов, если там тоже используются first_name/last_name.

2. Исправить ответственного за лида

Сейчас на /sales ответственный отображается как “—”.

Реальная таблица leads использует поле:

assigned_to

Нужно:
- использовать assigned_to как ответственного;
- фильтр “Все агенты” должен фильтровать по assigned_to;
- отображать имя агента через agents.id = leads.assigned_to;
- при создании лида сохранять assigned_to;
- при редактировании лида обновлять assigned_to.

Важно:
Сейчас в базе assigned_to может иметь внешний ключ не на agents.id. Если assigned_to не может ссылаться на agents.id, нужно исправить Supabase schema:
- либо foreign key leads.assigned_to → agents.id,
- либо добавить правильное поле lead_agent_id/agent_id и привести frontend/backend к нему.

Главное: ответственный в CRM должен реально быть агентом клиники и отображаться в /sales.

3. Исправить создание нового лида

Нельзя вставлять в leads поля:

first_name
last_name
agent_id

Потому что в реальной таблице их нет или они не используются.

Правильный insert/update должен использовать:

{
  clinic_id,
  full_name,
  phone,
  email,
  company,
  age,
  source,
  status_id,
  assigned_to,
  comment,
  service_id
}

После создания новый лид должен сразу появляться в списке /sales с именем, статусом и ответственным.

4. Исправить /admin → Статусы

Сейчас в базе есть:
- lead_statuses
- booking_statuses

Но вкладка /admin → Статусы показывает:
- “Нет статусов записей”
- “Нет статусов лидов”

Реальное поле сортировки в базе:

sort_order

Нужно полностью убрать использование:

position

И заменить на:

sort_order

Для:
- чтения lead_statuses;
- чтения booking_statuses;
- создания статуса;
- редактирования статуса;
- сортировки статусов;
- создания стандартных статусов.

Проверить:
- /admin → Статусы показывает все статусы лидов;
- /admin → Статусы показывает все статусы записей;
- созданный статус появляется сразу;
- статус можно редактировать;
- статус можно удалить.

5. Исправить отображение статусов в /sales

У каждого лида на /sales должен отображаться статус из:

lead_statuses.name

Связь:

leads.status_id → lead_statuses.id

Проверить:
- статус виден в таблице лидов;
- фильтр по статусу работает;
- при изменении статуса лида UI обновляется.

6. Исправить отображение ролей у агентов

Сейчас /admin → Агенты показывает роль “—”, хотя у агентов есть role_id.

Нужно:
- загрузить roles для clinic_id;
- отображать roles.name по agents.role_id;
- при создании агента выбирать роль;
- при редактировании агента выбирать роль;
- сохранять role_id в agents.

7. Исправить отображение прав в /admin → Роли

Сейчас роли отображаются, но права показываются как “Нет прав”, хотя permissions в базе заполнены.

Нужно привести permissions к одному формату.

В базе permissions может быть таким:

{
  "dashboard": true,
  "booking": true,
  "reception": true,
  "crm": true,
  "admin": true,
  "reports": true,
  "ads": true,
  "settings": true
}

UI должен читать именно эти ключи и показывать понятные названия:
- dashboard → Дашборд
- booking → Запись
- reception → Ресепшн
- crm → CRM
- admin → Админ
- reports → Отчёты
- ads → Реклама
- settings → Настройки

При создании/редактировании роли сохранять permissions в таком же формате.

8. Проверить /reception

Ресепшн работает, но после фиксов нужно повторно проверить:
- записи видны по датам;
- отображается клиент;
- отображается услуга;
- отображается агент;
- кнопка “Пришёл” сохраняет visited=true;
- кнопка “Не пришёл” сохраняет visited=false;
- связанный лид не удаляется;
- booking.lead_id остаётся заполненным.

9. Проверить /dashboard

Dashboard должен:
- показывать всех агентов;
- считать записи по каждому агенту;
- не падать на 1000+ записей;
- не зависать при 25+ агентах.

Создать или использовать тестовую клинику и проверить минимум:
- 25 агентов

Ожидаемый результат:

- /admin → Статусы показывает статусы;
- /admin → Роли показывает права;
- /admin → Агенты показывает роли;
