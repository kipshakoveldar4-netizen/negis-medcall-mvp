Нужно провести и исправить краш-тест Negis CRM.

Контекст:
Есть Negis Control Admin: https://admin.negis.online
Есть основной CRM сайт: https://www.negis.online
Админка умеет генерировать вход в клинику через impersonation token.
После входа из админки CRM открывается, но режим должен работать как полноценный доступ администратора клиники.

Задача:
1. Проверь вход в CRM из Negis Control по impersonation token.
2. После успешной проверки токена CRM должна открыть клинику без запроса логина/пароля.
3. В этом режиме должны работать реальные данные клиники:
   - сотрудники,
   - услуги/процедуры,
   - записи,
   - расписание,
   - создание новой записи,
   - редактирование данных, если это доступно админу клиники.
4. Сейчас проблема: при входе из админки CRM открывается, но записи могут не отображаться и создание записи работает не так, как при обычном логине клиники. Нужно исправить.

Что важно:
- Impersonation mode не должен быть просто визуальным режимом просмотра.
- После проверки токена от Negis Control CRM должна получить `clinicId` и работать с данными этой клиники.
- Если Supabase RLS/Auth мешает, нужно сделать backend endpoint в CRM, который работает от имени клиники безопасно, либо создать корректную сессию/контекст клиники.
- Нельзя хранить service role key на frontend.
- Service role key может быть только на backend/server side.
- Frontend должен обращаться только к безопасным API endpoint'ам.

Проверка:
Создать тестовую клинику или использовать существующую.
Создать:
- 10 сотрудников,
- 10 процедур/услуг,
- 10 разных отделов или категорий,
- 10 записей, чтобы клиент попал в каждый отдел.

Потом проверить два сценария:

Сценарий A: обычный вход в CRM под логином клиники
- сотрудники отображаются,
- услуги отображаются,
- записи отображаются,
- новая запись создается,
- слот в расписании обновляется.

Сценарий B: вход из Negis Control по кнопке “Войти”
- CRM открывается без логина/пароля,
- отображается та же клиника,
- сотрудники отображаются,
- услуги отображаются,
- записи отображаются,
- новая запись создается,
- слот в расписании обновляется.

Ожидаемый результат:
Сценарий B должен работать так же стабильно, как обычный вход под клиникой.

Дополнительно проверить:
- кириллица в именах сотрудников и названиях процедур сохраняется нормально;
- нет ошибки “Доступ по ссылке истёк”, если токен свежий;
- если токен реально истек, показывать понятное сообщение;
- после выхода из режима входа через админку очищать impersonation token/session;
- не показывать демоданные, если есть реальные данные клиники.
